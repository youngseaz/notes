# RFC 9114 解析：HTTP/3 协议分析

## 1. 概述
RFC 9114 定义了 HTTP/3 在 QUIC 之上的语义与消息交换方式。
HTTP/3 将传统 HTTP/2 的消息语义映射到 QUIC 流和帧结构，不再使用 TCP 连接，而是依赖 QUIC 的报文多路复用、流控、丢包恢复和加密。

HTTP/3 的核心数据结构包括：
- QUIC 流类型（Control Stream、Push Stream、QPACK Encoder/Decoder Stream、请求/响应双向流）
- HTTP/3 帧（HEADERS、DATA、SETTINGS、GOAWAY、PUSH_PROMISE 等）
- QPACK 头部压缩协商与流指令

## 2. HTTP/3 的流模型
HTTP/3 使用 QUIC 流来封装 HTTP 消息。RFC 9114 通过特殊的流类型标识初始控制流和编码/解码辅助流。

### 2.1 流类型值
- `0x00`：Control Stream（控制流）
- `0x01`：Push Stream（推送流）
- `0x02`：QPACK Encoder Stream（QPACK 编码器流）
- `0x03`：QPACK Decoder Stream（QPACK 解码器流）

这些流类型是单向流，且每个端点必须至少打开一个控制流。客户端和服务器都必须各自打开自己的控制流。

### 2.2 HTTP/3 帧头
HTTP/3 帧由类型和值长度构成，后面跟着帧负载。

```c
typedef struct {
    uint64_t type;   // 变长整数（frame type）
    uint64_t length; // 变长整数（payload length）
} http3_frame_header_t;
```

帧负载的内容取决于 `type`。

## 3. 连接建立与第一步交换
HTTP/3 依赖 QUIC 完成连接建立。RFC 9114 本身不定义加密握手，而是基于 QUIC 的 1-RTT 安全信道。

### 3.1 QUIC 建立完成后
一旦 QUIC 连接进入 `HANDSHAKE_COMPLETED`：
- 客户端打开一个 Control Stream
- 服务器打开一个 Control Stream
- 双方打开 QPACK Encoder Stream 和 QPACK Decoder Stream（可选地为推送建立 Push Stream）

控制流的第一个字节是流类型。

```c
typedef enum {
    HTTP3_STREAM_TYPE_CONTROL = 0x00,
    HTTP3_STREAM_TYPE_PUSH = 0x01,
    HTTP3_STREAM_TYPE_QPACK_ENCODER = 0x02,
    HTTP3_STREAM_TYPE_QPACK_DECODER = 0x03,
} http3_stream_type_t;

typedef struct {
    uint64_t stream_type; // 变长整数
} http3_stream_type_frame_t;
```

### 3.2 Control Stream 首帧：SETTINGS
客户端和服务器必须在自己的 Control Stream 中发送一个 `SETTINGS` 帧，表示 HTTP/3 协议参数。

```c
typedef struct {
    uint64_t identifier; // 设置项 ID
    uint64_t value;      // 设置项值
} http3_settings_parameter_t;

typedef struct {
    http3_frame_header_t header; // type = 0x4
    http3_settings_parameter_t *params;
    size_t param_count;
} http3_settings_frame_t;
```

常见设置项：
- `0x1`：QPACK_MAX_TABLE_CAPACITY
- `0x7`：QPACK_BLOCKED_STREAMS

示例：
```c
http3_settings_parameter_t client_settings[] = {
    {0x1, 0x2000}, // QPACK_MAX_TABLE_CAPACITY = 8192
    {0x7, 0x10},   // QPACK_BLOCKED_STREAMS = 16
};
```

## 4. QPACK 流建立与协商
HTTP/3 使用 QPACK 作为头部压缩协议。

### 4.1 QPACK Encoder Stream
客户端必须打开一个 QPACK Encoder Stream，并发送编码器指令。

```c
typedef struct {
    uint64_t instruction_type; // qpack instruction type
    uint64_t data_length;
    uint8_t *data;
} qpack_instruction_t;
```

### 4.2 QPACK Decoder Stream
服务器必须打开一个 QPACK Decoder Stream，并发送解码器指令。

```c
typedef struct {
    uint64_t instruction_type; // qpack decoder instruction type
    uint64_t data_length;
    uint8_t *data;
} qpack_decoder_instruction_t;
```

这两个辅助流不包含 HTTP/3 帧头，而是直接发送 QPACK 指令流。

## 5. HTTP 请求 / 响应的每一步数据结构
HTTP/3 请求和响应在双向流上发生。客户端发起的请求使用客户端发起的双向流，流 ID 为奇数。

### 5.1 打开请求流
客户端创建一个新的双向流：
```
stream_id = 0x... (client-initiated bidirectional, odd)
```

### 5.2 发送 HEADERS 帧
请求头通过 HEADERS 帧发送，帧类型为 `0x1`。

```c
typedef struct {
    http3_frame_header_t header; // type = 0x1
    uint8_t *header_block;       // QPACK 头块编码后的数据
    size_t header_block_length;
} http3_headers_frame_t;
```

请求头部示例：
- `:method: GET`
- `:scheme: https`
- `:authority: example.com`
- `:path: /index.html`

HEADERS 帧负载是 QPACK 编码后的字节流。

### 5.3 发送 DATA 帧
请求正文使用 `DATA` 帧发送，帧类型为 `0x0`。

```c
typedef struct {
    http3_frame_header_t header; // type = 0x0
    uint8_t *data;
    size_t data_length;
} http3_data_frame_t;
```

当 `DATA` 帧的负载长度为 0 时，表示空数据帧；流关闭由 QUIC STREAM FIN 标志表示，而不是 HTTP/3 终止帧。

### 5.4 服务器响应 HEADERS
服务器在相同请求流上发送 `HEADERS` 帧作为响应元信息。该帧结构与请求 HEADERS 相同。

### 5.5 服务器响应 DATA
响应数据通过一系列 `DATA` 帧发送：

```c
http3_data_frame_t response_data = {
    .header = {.type = 0x0, .length = payload_length},
    .data = payload,
    .data_length = payload_length,
};
```

### 5.6 结束流
响应结束时，服务器可以发送带 FIN 的最后一个 QUIC 流帧，或者在最后一个 `DATA` 帧后关闭流。HTTP/3 不要求特殊结束帧。

## 6. 控制帧与连接操作
HTTP/3 定义的控制帧可用于流级和连接级通知。

### 6.1 CANCEL_PUSH
客户端可发送 `CANCEL_PUSH`（帧类型 `0x3`）取消先前的推送。

```c
typedef struct {
    http3_frame_header_t header; // type = 0x3
    uint64_t push_id;
} http3_cancel_push_frame_t;
```

### 6.2 PUSH_PROMISE
服务器在客户端请求流上发送 `PUSH_PROMISE`（帧类型 `0x5`），表示推送响应。

```c
typedef struct {
    http3_frame_header_t header; // type = 0x5
    uint64_t push_id;
    uint8_t *header_block;
    size_t header_block_length;
} http3_push_promise_frame_t;
```

推送流仍然使用独立的 Push Stream，但 `PUSH_PROMISE` 提供了推送标识和头部元信息。

### 6.3 GOAWAY
当服务器准备关闭连接时，发送 `GOAWAY`（帧类型 `0x7`），告知客户端不再接收新的请求流。

```c
typedef struct {
    http3_frame_header_t header; // type = 0x7
    uint64_t stream_id;
} http3_goaway_frame_t;
```

该 `stream_id` 表示服务器仅接受小于该值的客户端请求流。

### 6.4 MAX_PUSH_ID
服务器使用 `MAX_PUSH_ID`（帧类型 `0xd`）通告客户允许的最大推送 ID。

```c
typedef struct {
    http3_frame_header_t header; // type = 0xd
    uint64_t max_push_id;
} http3_max_push_id_frame_t;
```

## 7. HTTP/3 数据交换流程步骤
下面通过步骤说明 HTTP/3 的典型交互。

### 步骤 1：QUIC 连接与 TLS 完成
- QUIC 建立完成
- TLS 握手完成
- 进入 1-RTT 数据阶段

在这个阶段，HTTP/3 数据可通过 QUIC 1-RTT 加密传输。

### 步骤 2：打开控制流并发送 SETTINGS
- 客户端打开 Control Stream，发送流类型 `0x00`
- 客户端发送 `SETTINGS` 帧
- 服务器打开 Control Stream，发送流类型 `0x00`
- 服务器发送 `SETTINGS` 帧

### 步骤 3：建立 QPACK 辅助流
- 客户端打开 QPACK Encoder Stream，发送流类型 `0x02`
- 客户端打开 QPACK Decoder Stream，发送流类型 `0x03`
- 服务器同样建立自己的 QPACK 流

### 步骤 4：创建请求流并发送 HEADERS
- 客户端打开一个新的双向流（client-initiated bidirectional）
- 在该流上发送 HEADERS 帧
- 如果有请求体，再发送 DATA 帧

### 步骤 5：服务器发送响应 HEADERS + DATA
- 服务器在相同流上发送 HEADERS
- 服务器发送若干 DATA 帧
- 服务器结束流

### 步骤 6：推送与取消
- 服务器若发起服务端推送，先发送 PUSH_PROMISE
- 客户端可发送 CANCEL_PUSH 或主动忽略
- 服务器可发送 MAX_PUSH_ID 控制推送许可

### 步骤 7：连接关闭
- 如果服务器不再接受新请求，发送 GOAWAY
- 双方继续完成已存在流
- QUIC 连接关闭由 QUIC 关闭帧负责

## 8. HTTP/3 关键帧类型与字节结构
HTTP/3 帧的帧头由 `type` 和 `length` 两个变长整数构成，后面接 `length` 字节的负载。下面补充常见帧类型的字段和字节级含义。

### 8.1 DATA 帧字节结构
DATA 帧结构：
- `type` = 0x0
- `length` = payload length
- `payload` = raw data bytes

```c
typedef struct {
    uint64_t type;   // 0x0
    uint64_t length; // payload length
    uint8_t *data;   // payload bytes
} http3_data_frame_t;
```

字节序示例（假设payload 5字节）：
- frame type: 0x00
- length: 0x05
- data: `48 65 6c 6c 6f`

### 8.2 HEADERS 帧字节结构
HEADERS 帧结构：
- `type` = 0x1
- `length` = header block length
- `payload` = QPACK 头块

```c
typedef struct {
    uint64_t type;              // 0x1
    uint64_t length;            // header block length
    uint8_t *header_block;      // QPACK 编码数据
    size_t header_block_length;
} http3_headers_frame_t;
```

HEADERS 帧的负载并非文本，而是 QPACK 编码后的字节。

### 8.3 SETTINGS 帧字节结构
SETTINGS 帧结构：
- `type` = 0x4
- `length` = settings payload length
- payload = 0或多个设置项，每个设置项为 `identifier` + `value`

```c
typedef struct {
    uint64_t type;   // 0x4
    uint64_t length; // payload length
    uint64_t identifier;
    uint64_t value;
} http3_settings_frame_t;
```

单个设置项例如：
- identifier = 0x1
- value = 0x2000

### 8.4 PUSH_PROMISE 帧字节结构
PUSH_PROMISE 帧结构：
- `type` = 0x5
- `length` = push id length + header block length
- `push_id` = 变长整数
- `header_block` = QPACK 头块

```c
typedef struct {
    uint64_t type;              // 0x5
    uint64_t length;
    uint64_t push_id;
    uint8_t *header_block;
    size_t header_block_length;
} http3_push_promise_frame_t;
```

### 8.5 GOAWAY 帧字节结构
GOAWAY 帧结构：
- `type` = 0x7
- `length` = stream_id length
- `stream_id` = 变长整数

```c
typedef struct {
    uint64_t type;   // 0x7
    uint64_t length; // variable size of stream_id
    uint64_t stream_id;
} http3_goaway_frame_t;
```

## 9. QPACK 与 HTTP/3 的具体交互
HTTP/3 的 HEADERS 负载是 QPACK 头块；QPACK 通过 Encoder/Decoder 辅助流同步动态表状态。下面补充典型交换流程。

### 9.1 QPACK 编码器流指令示例
QPACK Encoder Stream 指令包括：
- `Insert With Name Reference`
- `Insert Without Name Reference`
- `Duplicate`

```c
typedef enum {
    QPACK_INS_INSERT_WITH_NAME_REF = 0x00,
    QPACK_INS_INSERT_WITHOUT_NAME_REF = 0x40,
    QPACK_INS_DUPLICATE = 0x80,
} qpack_encoder_instruction_type_t;
```

指令结构示例：
```c
typedef struct {
    uint64_t instruction_type;
    uint64_t name_index;
    uint64_t value_length;
    uint8_t *value;
} qpack_insert_with_name_ref_t;
```

### 9.2 QPACK 解码器流指令示例
QPACK Decoder Stream 指令包括：
- `Header Acknowledgement`
- `Stream Cancellation`
- `Insert Count Increment`

```c
typedef enum {
    QPACK_DEC_INSTR_HEADER_ACK = 0x00,
    QPACK_DEC_INSTR_CANCEL_STREAM = 0x01,
    QPACK_DEC_INSTR_INSERT_COUNT_INCREMENT = 0x02,
} qpack_decoder_instruction_type_t;
```

示例结构：
```c
typedef struct {
    uint64_t instruction_type;
    uint64_t stream_id;
} qpack_decoder_cancel_stream_t;
```

### 9.3 HTTP/3 HEADERS / QPACK 交互流程
1. 客户端打开 QPACK Encoder/Decoder Stream。
2. 客户端在 Control Stream 上发送 SETTINGS，包含 QPACK 参数。
3. 服务器在自己的 QPACK Encoder/Decoder Stream 上发送动态表指令。
4. 客户端在请求流上发送 HEADERS，头块可引用动态表条目。
5. 服务器在解码 HEADERS 后，发送 `Header Acknowledgement`，并根据需要发送 `Insert Count Increment`。

这意味着 HTTP/3 的 HEADERS 并非独立可解码数据，而是依赖于 QPACK 动态表状态。

## 10. HTTP/3 内部结构和状态管理
HTTP/3 本身并不定义完整状态机，但可以抽象为以下几个层次结构。

### 10.1 HTTP/3 连接状态
```c
typedef enum {
    HTTP3_CONN_STATE_IDLE,
    HTTP3_CONN_STATE_READY,
    HTTP3_CONN_STATE_GOAWAY_SENT,
    HTTP3_CONN_STATE_CLOSING,
    HTTP3_CONN_STATE_DRAINING,
} http3_connection_state_t;
```

### 10.2 HTTP/3 流状态
```c
typedef enum {
    HTTP3_STREAM_STATE_IDLE,
    HTTP3_STREAM_STATE_OPEN,
    HTTP3_STREAM_STATE_HALF_CLOSED_LOCAL,
    HTTP3_STREAM_STATE_HALF_CLOSED_REMOTE,
    HTTP3_STREAM_STATE_CLOSED,

typedef struct {
    uint64_t stream_id;
    http3_stream_state_t state;
    bool has_headers;
    bool has_data;
} http3_stream_context_t;
```

### 10.3 HTTP/3 控制流上下文
```c
typedef struct {
    bool settings_received;
    bool qpack_encoder_open;
    bool qpack_decoder_open;
    uint64_t max_push_id;
} http3_control_context_t;
```

## 11. 扩展：推送与取消流程详解
### 11.1 服务器推送流程
1. 服务器在 Control Stream 发送 SETTINGS（若未发送）。
2. 服务器在请求流上发送 `PUSH_PROMISE` 帧。
3. 客户端收到 `PUSH_PROMISE` 后，可以选择接受或取消。
4. 如果取消，客户端在请求流上发送 `CANCEL_PUSH`。如果接受，客户端使用新的 Push Stream 消耗推送内容。

### 11.2 取消推送的数据结构
```c
typedef struct {
    uint64_t type;   // 0x3
    uint64_t length; // payload length
    uint64_t push_id;
} http3_cancel_push_frame_t;
```

### 11.3 MAX_PUSH_ID 的作用
`MAX_PUSH_ID` 控制服务器可创建的最大推送 ID。应用场景：客户端限制服务器推送数量以避免资源耗尽。

## 13. HTTP/3 交互流程扩展
下面补充更细化的交互流程，强调每个阶段的报文序列、流类型和状态转换。

### 13.1 连接初始化与 Control/QPACK 流并行建立
1. QUIC 和 TLS 完成后，客户端立即创建自己的 Control Stream：单向流，流头是 `0x00`。
2. 客户端发送 `SETTINGS` 帧，通告 QPACK 参数与其他 HTTP/3 设置。
3. 客户端打开 QPACK Encoder Stream（`0x02`）和 QPACK Decoder Stream（`0x03`）。
4. 服务器检测到客户端的 Control/QPACK 流，建立自己的 Control Stream、Encoder Stream、Decoder Stream。
5. 服务器发送 `SETTINGS` 帧，完成初始协商。

> 这一步是 HTTP/3 应用层的“握手”，它与 QUIC/TLS 握手平行发生，但必须先完成才能安全交换 HEADERS。

### 13.2 请求流的创建与请求报文发送
1. 客户端创建一个新的双向流 `stream_id = 0x1`（第一个客户端发起双向流）。
2. 在该流上发送 `HEADERS` 帧，负载为 QPACK 编码的头块。
3. 如果请求包含主体，则紧随其后发送一个或多个 `DATA` 帧。
4. 客户端关闭发送方向，并由 QUIC 的 STREAM FIN 标志表示请求结束。

流程示例：
- `stream_id 0x1` open
- `HEADERS` frame(type=0x1, length=N)
- `DATA` frame(type=0x0, length=M) [可选]
- QUIC FIN

### 13.3 服务器响应的生成与流响应序列
1. 服务器在同一请求流上发送 `HEADERS` 帧作为响应首部。
2. 服务器根据需要发送 `DATA` 帧，可能分段传输多个 DATA 帧。
3. 服务器在最后一个 DATA 帧末尾或单独 FIN 信号上关闭流。

响应示例：
- `HEADERS` frame(type=0x1, length=P)
- `DATA` frame(type=0x0, length=Q)
- `DATA` frame(type=0x0, length=R) [可选]
- QUIC FIN

### 13.4 并发请求与多路复用
- 客户端可同时打开多个请求流：`0x1`, `0x5`, `0x9`, ...
- 每个请求流独立传输 HEADERS/DATA，服务器并行响应
- QUIC 负责底层流控与拥塞控制，HTTP/3 不再复用一个连接上的“消息边界”

这种并发模型是 HTTP/3 性能提升的核心：请求、响应和控制流可并行进行，避免队头阻塞。

### 13.5 Control Stream 事件与异步通知
Control Stream 用于发送控制帧，典型交互包括：
- `SETTINGS`：连接参数协商
- `GOAWAY`：服务器拒绝新请求
- `MAX_PUSH_ID`：控制允许的推送数量

典型顺序：
- 服务器在准备关闭连接时发送 `GOAWAY`（type=0x7）
- 客户端收到后停止创建新的请求流，但继续完成已存在流
- 服务器可继续通过已存在流发送响应

### 13.6 推送处理与取消流程
1. 服务器在已有请求流上发送 `PUSH_PROMISE`，声明一个推送 ID 和头块。
2. 该推送对应一个新的服务器单向 Push Stream，其流头是 `0x01`。
3. 客户端收到 `PUSH_PROMISE` 后可选择消费或取消推送。
4. 若取消，客户端发送 `CANCEL_PUSH` 帧；若消费，则在新的 Push Stream 上接收 HEADERS 和 DATA。

完整序列：
- 请求流：`PUSH_PROMISE`(push_id)
- Push Stream：`0x01` open
- Push Stream：`HEADERS` + `DATA` + FIN
- 若取消：请求流 `CANCEL_PUSH`(push_id)

### 13.7 QPACK 头块解码同步详细流程
1. 客户端/服务器因 SETTINGS 协商 QPACK 参数。
2. Encoder Stream 发送动态表插入指令。
3. Decoder Stream 发送 `Insert Count Increment` 和 `Header Acknowledgement`。
4. HEADERS 帧中的头块引用动态表条目或直接编码字面值。
5. 服务器/客户端收到 HEADERS 后使用本地动态表解码。

> 这意味着 HEADERS 不能独立于 QPACK 动态表状态解码；它依赖于先前或并行的辅助流同步。

### 13.8 HTTP/3 中的错误通知与连接关闭
HTTP/3 的应用错误通常由 QUIC 错误码承载。常见模式：
- 客户端发送非法帧类型或格式错误 → QUIC 终止连接
- 服务器接收非法 HEADERS 或 QPACK 同步错误 → 发送 CONNECTION_CLOSE
- `GOAWAY` 只表示拒绝新请求，不关闭已存在流

```c
typedef struct {
    uint64_t error_code;
    uint64_t frame_type;
    uint64_t reason_length;
    uint8_t *reason_phrase;
} http3_connection_close_t;
```

### 13.9 连接回收与 Draining 状态
1. 服务器发送 `GOAWAY`。
2. 客户端停止创建新的请求流。
3. 服务器继续完成已有流的响应。
4. QUIC 进入 `DRAINING` 状态后退出连接。

这种设计保证了 HTTP/3 能优雅关闭连接，同时让 TCP/TLS 阶段的“优雅关闭”概念转移到 QUIC 的流状态管理上。

## 15. HEADERS/QPACK 字节级示例
为了让 RFC 9114 的交互更具体，这里补一个简化的 HEADERS 负载示例。

### 15.1 QPACK 头块结构
HTTP/3 HEADERS 帧负载为 QPACK 头块，其常见格式包含：
- Required Insert Count（变长整数）
- Base（变长整数）
- Header Field 表示（静态表索引、动态表索引或文字字面值）

```c
typedef struct {
    uint64_t required_insert_count;
    uint64_t base;
    uint8_t *block;
    size_t block_length;
} qpack_header_block_t;
```

### 15.2 简化示例：请求 HEADERS
假设请求头为：
- `:method: GET`
- `:scheme: https`
- `:authority: example.com`
- `:path: /index.html`

一个可能的简化 QPACK 头块示例（非完整精确编码，但反映结构）：
- Required Insert Count = 0x00
- Base = 0x00
- `:method` = 静态表索引 0x82（表示 `GET`）
- `:scheme` = 静态表索引 0x84（表示 `https`）
- `:authority` = 字面名称 + 字面值
- `:path` = 静态表索引 0x86（表示 `/index.html`）

HEADERS 帧字节序列示例：
- frame type: `0x01`
- length: `0x0b`（假设头块总长 11 字节）
- payload: `00 00 82 84 40 0b 6578616d706c652e636f6d 86`

其中 `40` 表示插入字面名称/值对，`0b` 为 `example.com` 长度。

### 15.3 Control Stream 字节例子
Control Stream 数据从流类型开始：
- `0x00`（Control Stream）
- SETTINGS 帧头：`0x04`（type），`0x04`（length）
- 设置项：`0x01` `0x20` `0x00`（QPACK_MAX_TABLE_CAPACITY = 8192）

```c
uint8_t control_stream_example[] = {
    0x00,       // stream type = Control
    0x04, 0x04, // SETTINGS frame header (type=4, length=4)
    0x01, 0x20, 0x00,
};
```

### 15.4 请求流字节例子
请求流的字节序列示例：
- HEADERS：`0x01` `0x0b` `...header block...`
- DATA：`0x00` `0x05` `48 65 6c 6c 6f`

```c
uint8_t request_stream_example[] = {
    0x01, 0x0b, 0x00, 0x00, 0x82, 0x84, 0x40, 0x0b,
    'e','x','a','m','p','l','e','.','c','o','m',
    0x00, 0x05, 'H','e','l','l','o',
};
```

> 注意：在 QUIC 中，这些字节被封装在 STREAM 帧内，且可能与其他流的 STREAM 帧混合在同一个 QUIC 包里。

## 16. HTTP/3 与 QUIC 报文时间线示例
下面给出一个典型交互中的 QUIC 报文序列视角。

### 16.1 客户端发送阶段
1. QUIC 1-RTT 包 A：包含 Control Stream 的 `SETTINGS` STREAM 帧。
2. QUIC 1-RTT 包 B：包含 QPACK Encoder Stream 的指令。
3. QUIC 1-RTT 包 C：包含请求流 `0x1` 的 HEADERS STREAM 帧。
4. QUIC 1-RTT 包 D：包含请求流 `0x1` 的 DATA STREAM 帧并携带 FIN。

### 16.2 服务器响应阶段
1. QUIC 1-RTT 包 E：包含服务器 Control Stream 的 `SETTINGS`。
2. QUIC 1-RTT 包 F：包含服务器请求流 `0x1` 的响应 HEADERS。
3. QUIC 1-RTT 包 G：包含服务器请求流 `0x1` 的响应 DATA 并携带 FIN。

### 16.3 多路复用示意
- 同一 QUIC 包可以包含多个 STREAM 帧，如 Control Stream + 请求流 HEADERS。
- 不同 HTTP/3 流之间通过 QUIC 数据流 ID 区分。
- 这使得 HTTP/3 可以在单个 QUIC 连接上并发传输多个请求/响应对。

## 17. HTTP/3 中的报文与状态变化摘要
### 17.1 从应用层看流程
- HTTP/3 应用层打开 Control/QPACK 流，建立协议环境。
- 客户端打开请求流并发送 HEADERS、DATA。
- 服务器发送响应 HEADERS、DATA 并关闭流。
- Control Stream 处理连接级事件，如 GOAWAY、MAX_PUSH_ID。

### 17.2 关键状态机与流事件
```c
typedef struct {
    uint64_t stream_id;
    http3_stream_state_t state;
    bool headers_sent;
    bool data_sent;
    bool fin_sent;
    bool headers_received;
    bool data_received;
    bool fin_received;
} http3_stream_event_t;
```

每个 HTTP/3 流在接收到：
- HEADERS → 进入 `OPEN` 并标记 `headers_received`
- DATA → 累积数据
- FIN → 进入 `CLOSED`

### 17.3 连接收束
- `GOAWAY` 触发客户端停止创建新请求流
- 已存在流继续完成
- QUIC 最终发送 CONNECTION_CLOSE 或 PATH_CLOSE
