## RFC9000 报文格式总结

### 1. QUIC 包的基本结构
- QUIC 包封装在 UDP 数据报中，一个 UDP 数据报可以携带一个或多个 QUIC 包。
- QUIC 包分为两类：长头部（Long Header）和短头部（Short Header）。
- 包内载荷由帧（Frame）构成，帧负责传输控制信息和流数据。
- 所有 QUIC 包都使用变长整数编码（Variable Length Integer），以减小头部开销。

### 2. 长头部包（握手阶段）
- Header Form = 1，包含版本号和连接 ID 信息。
- 公共字段：
  - 1 字节标志（Header Form + Fixed bit + Long Packet Type + Reserved bits + Packet Number Length）
  - Version（32 bit）
  - DCID Length（8 bit） + Destination Connection ID
  - SCID Length（8 bit） + Source Connection ID
- 长头部包后续包含：
  - Token Length（可变整数） + Token
  - Length（可变整数）：剩余负载长度
  - Packet Number（1-4 字节）
  - Payload：对称加密后的数据
- 常见长头部包类型：
  - Initial：客户端发起连接，携带初始 TLS 握手数据和 CRYPTO 帧；必须至少填充到 1200 字节。
  - 0-RTT：客户端在已有密钥材料时发送，允许早期应用数据；不可用于地址验证、重传或路径迁移失败恢复。
  - Handshake：客户端/服务器交换 TLS 握手记录，该阶段的消息使用 Handshake keys 保护。
  - Retry：服务器可选回应；用于地址验证。包中不包含 Packet Number，而是包含 Retry Token 和完整性标签。
  - Version Negotiation：服务器回应不支持的版本。包含 server-chosen versions 列表；不加密且不包含 Packet Number。

### 3. 短头部包（1-RTT 阶段）
- Header Form = 0，头部更短，仅用于已完成握手后的 1-RTT 数据传输。
- 结构：
  - 1 字节标志（Header Form = 0 + Spin bit + Reserved bits + Key Phase + Packet Number Length）
  - Destination Connection ID
  - Packet Number（1-4 字节，截断编码）
  - Payload：加密后的 1-RTT 数据
- 1-RTT 包可携带任意合法的 QUIC 帧，并且帧也被加密和认证。

### 4. 每种报文类型的字段详细说明
#### 4.1 Initial 包
- 目标：发起连接并开始 TLS 握手。
- 关键字段：
  - Token Length：初始为 0；若收到 Retry 包则为 Retry Token 长度。
  - Token：地址验证令牌。
  - Length：剩余加密负载的总长度（含 Packet Number）。
  - Packet Number：数据包序号，进行丢失检测和重传。
  - Payload：包含 CRYPTO 帧和可能的 Stream 数据帧。
- 特点：必须至少 1200 字节长，防止放大攻击。

#### 4.2 Retry 包
- 目标：验证客户端地址并防止欺骗。
- 关键字段：
  - 原始 DCID（Original Destination Connection ID）：指回客户端初始发送的 DCID。
  - SCID：包含服务器的源 Connection ID。
  - Retry Token：客户端必须在后续 Initial 包中回传。
  - Integrity Tag：对 Retry 包进行完整性保护。
- 特点：不包含 Packet Number，不能重传。

#### 4.3 Handshake 包
- 目标：交换完成握手所需的 TLS 记录。
- 关键字段：
  - 与 Initial 包类似的长头部结构。
  - Payload：加密的 CRYPTO 帧，携带 TLS 握手消息。
- 特点：Handshake 包与 Initial 包使用不同的加密密钥。

#### 4.4 0-RTT 包
- 目标：客户端在握手完成前尽早发送应用数据。
- 关键字段：
  - 与 Initial 包相同的长头部结构。
  - Payload：以 0-RTT keys 加密的 STREAM 或 PING 等帧。
- 安全约束：只能发送幂等数据或允许重放的数据，因重放攻击风险。

#### 4.5 1-RTT 包
- 目标：正常数据传输阶段的所有业务数据包。
- 关键字段：
  - 短头部标志
  - Destination Connection ID
  - Packet Number
  - Payload：任意 1-RTT 帧
- 特点：头部最小，适合高效传输。

### 5. 关键帧格式
#### 5.1 STREAM 帧
- 用于传输应用层字节流。
- 字段：Frame Type + Stream ID + Offset（可选）+ Length（可选）+ Data
- 其中 Stream ID 和 Offset 使用可变整数编码。

#### 5.2 ACK 帧
- 用于确认接收的包号。
- 字段：Frame Type + Largest Acknowledged + ACK Delay + ACK Range Count + First ACK Range + [ACK Range Lengths...]
- 可选 ECN Counts：ECT(0)、ECT(1)、CE，用于 ECN 报告。

#### 5.3 CRYPTO 帧
- 承载 TLS 握手消息。
- 字段：Frame Type + Offset + Length + Crypto Data
- 用于 Initial、Handshake、0-RTT 阶段。

#### 5.4 CONNECTION_CLOSE 帧
- 用于终止连接。
- 字段：Frame Type + Error Code + Frame Type（可选）+ Reason Length + Reason Phrase
- 连接错误或应用错误均通过该帧传递。

#### 5.5 PATH_CHALLENGE / PATH_RESPONSE
- PATH_CHALLENGE：8 字节随机数据；用于路径验证。
- PATH_RESPONSE：回送同样的 8 字节数据，证明路由可达。

#### 5.6 NEW_TOKEN 帧
- 服务器向客户端发送地址验证 Token。
- 字段：Frame Type + Token Length + Token

### 6. 握手流程的“每一步报文格式”补充说明
- 客户端 Initial：长头部 + Token Length=0（或携带 Retry Token）+ Packet Number + CRYPTO 数据。此报文用于开始 TLS 握手和传输第一个 QUIC 包。
- 服务器 Retry：长头部 + 原始 DCID + Retry Token + Integrity Tag。服务器在地址验证需求时发出，客户端必须重新发送 Initial。
- 服务器 Handshake：长头部 + Packet Number + 加密 CRYPTO 数据。用于继续 TLS 握手并在握手中期携带服务器证书和密钥信息。
- 客户端 0-RTT：长头部 + Packet Number + 0-RTT 加密的 STREAM 数据。允许客户端提前发送应用数据，但需防范重放。
- 1-RTT：短头部 + Packet Number + 受保护的帧。握手完成后，所有普通流量都走该类型。

### 7. 报文场景总结
- 连接建立阶段
  - Client Initial → Server Retry（可选）→ Server Initial/Handshake → Client Handshake/0-RTT → Server Handshake
- 数据传输阶段
  - 1-RTT 包携带 STREAM + ACK + 控制帧
- 地址验证与路径检测
  - Retry、NEW_TOKEN、PATH_CHALLENGE、PATH_RESPONSE
- 连接关闭阶段
  - CONNECTION_CLOSE 载入错误码和原因

### 8. RFC9000 协议交互流程
#### 8.1 连接初始化与握手
1. 客户端发送 `Initial` 包：
   - 长头部 + `Version` + `DCID/SCID`
   - 包含 `CRYPTO` 帧，携带 TLS ClientHello
   - Token Length 初始为 0
2. 服务器可能返回 `Retry` 包：
   - 用于地址验证
   - 包含 `Retry Token` 和 `Original Destination Connection ID`
   - 客户端接收后重发 `Initial` 包，并携带该 Token
3. 服务器发送 `Handshake` 包：
   - 包含 TLS ServerHello、证书、密钥交换等消息
   - 使用 Handshake keys 加密
4. 客户端发送 `Handshake` 包：
   - 完成 TLS 握手的后续消息，如 Finished
   - 可同时发送 `0-RTT` 包（如果客户端已有早期密钥材料）
5. 服务器发送最后 `Handshake` 包：
   - 发送 TLS 完成消息
   - 握手完成后，双方均可开始使用 1-RTT 保护的数据包

#### 8.2 0-RTT 与早期数据
- 客户端在以前会话基础上可以发送 `0-RTT` 包
- 0-RTT 数据包含 `STREAM` 帧或 `PING`
- 此前期数据由 0-RTT keys 保护，可能被重放
- 服务器必须对 0-RTT 数据进行重放安全性检查

#### 8.3 1-RTT 数据传输阶段
- 握手完成后，双方使用 `Short Header` 的 1-RTT 包进行正常数据传输
- 1-RTT 包可承载：
  - `STREAM` 帧（应用数据）
  - `ACK` 帧（确认包号）
  - `PING`、`PATH_CHALLENGE`、`PATH_RESPONSE`
  - 流控制和连接控制帧（如 `MAX_DATA`、`NEW_CONNECTION_ID`）
- 双方通过 ACK 和丢失检测实现可靠传输

#### 8.4 连接迁移与路径验证
- 客户端可以在 1-RTT 阶段迁移到新路径
- 新路径探测使用 `PATH_CHALLENGE` / `PATH_RESPONSE`
- 仅客户端可主动迁移，服务器也可使用 `Preferred Address`
- 迁移过程中，新的连接 ID 用于确保包与连接匹配

#### 8.5 连接终止
- 正常关闭：
  - 一方发送 `CONNECTION_CLOSE` 帧
  - 对端收到后回应 ACK，并进入 draining 状态
- 非正常关闭：
  - `CRYPTO` 或协议错误导致 `CONNECTION_CLOSE`
  - 可能触发 stateless reset

#### 8.6 协议交互关键点
- 握手与传输参数在 `CRYPTO` 帧中完成
- 长头部包用于连接建立和版本协商
- 短头部包用于握手完成后的高效数据传输
- ACK、丢失检测、流控制是 1-RTT 阶段的核心互动机制

#### 8.7 握手报文序列示例
1. `Client Initial`：客户端发送 Initial 包
   - 内容：`CRYPTO` 帧（ClientHello）
   - 目的：开始 TLS 握手，携带客户端 Connection ID
2. `Server Retry`（可选）：服务器返回 Retry 包
   - 内容：`Retry Token`，`Original DCID`
   - 目的：验证客户端源地址，防止 IP 欺骗
3. `Server Handshake`：服务器发送 Handshake 包
   - 内容：`CRYPTO` 帧（ServerHello、证书、Finished）
   - 目的：完成服务器端 TLS 协商
4. `Client Handshake`：客户端发送 Handshake 包
   - 内容：`CRYPTO` 帧（Finished）
   - 目的：完成客户端 TLS 协商
5. `Client 0-RTT`（可选）：客户端发送 0-RTT 包
   - 内容：`STREAM` 帧、`PING`
   - 目的：提前发送应用数据
6. `Server Handshake`：服务器确认握手完成
   - 内容：`CRYPTO` 帧（Finished）
   - 目的：握手结束，开始 1-RTT 数据传输

#### 8.8 连接迁移与路径验证序列
1. 客户端检测到新网络路径。
2. 客户端在新路径上发送 `PATH_CHALLENGE` 包。
3. 服务器返回 `PATH_RESPONSE`。
4. 客户端验证响应并启用新路径。
5. 服务器可通过 `NEW_CONNECTION_ID` 提供备用 Connection ID。

#### 8.9 丢失检测与重传交互
- 发送端通过 `Packet Number` 和 ACK 反馈判断丢包。
- 若未收到确认，触发重传：重新发送丢失的帧。
- `ACK` 包的 `ACK Range` 与 `ACK Delay` 表示确认状态。
- `PADDING` 与 `PING` 帧可以被用来触发 ACK，减少确认延迟。

#### 8.10 连接状态机简要描述
- `CLOSED` → `HANDSHAKE`：客户端发送 `Initial`。
- `HANDSHAKE` → `ESTABLISHED`：完成 TLS 握手，双方收到 `Finished`。
- `ESTABLISHED`：发送/接收 1-RTT 包，进行流控制和拥塞控制。
- `DRAINING`：一方发送 `CONNECTION_CLOSE` 后进入；继续处理已接收包，但不再发送新的应用数据。
- `CLOSED`：延迟一段时间后释放连接状态。

#### 8.11 关键交互注意事项
- `Initial` 包必须至少 1200 字节，防止路由器丢弃和放大攻击。
- `Retry` 包不可以重传；客户端必须重新生成新的 `Initial`。
- 只有在握手完成后才能使用 `Short Header` 的 1-RTT 包。
- 0-RTT 数据可能被重放，因此只能发送幂等或可安全重放的数据。

#### 8.12 握手协议交互示例
下面展示一个典型的 QUIC 握手交互流程，按报文和帧类型列出：

1. 客户端发送 `Initial` 包
   - 包类型：长头部，`Packet Type = Initial`
   - 头部字段：`Version`, `DCID`, `SCID`, `Token Length = 0`
   - 携带帧：`CRYPTO`（TLS ClientHello）

2. 服务器发送 `Retry` 包（可选）
   - 包类型：长头部，`Packet Type = Retry`
   - 头部字段：`Original DCID`, `SCID`, `Retry Token`
   - 作用：验证客户端地址，要求客户端重发 `Initial`

3. 客户端发送第二个 `Initial` 包
   - 头部字段：携带 `Retry Token`
   - 携带帧：`CRYPTO`（重新发送或继续 ClientHello）

4. 服务器发送 `Handshake` 包
   - 包类型：长头部，`Packet Type = Handshake`
   - 携带帧：`CRYPTO`（ServerHello、证书、ServerFinished）

5. 客户端发送 `0-RTT` 包（如果支持）
   - 包类型：长头部，`Packet Type = 0-RTT`
   - 携带帧：`STREAM`（早期应用数据）、`PING`
   - 同时发送 `Handshake` 包以完成握手

6. 服务器发送 `Handshake` 包
   - 携带帧：`CRYPTO`（ServerFinished）

7. 握手完成后，双方开始交换 `1-RTT` 包
   - 包类型：短头部
   - 携带帧：`STREAM`, `ACK`, `MAX_DATA`, `PATH_CHALLENGE`, `PATH_RESPONSE`

#### 8.13 示例协议状态机结构
一个简化的 QUIC 连接状态机可以表示为：

```c
typedef enum {
    QUIC_STATE_CLOSED,
    QUIC_STATE_INITIAL_SENT,
    QUIC_STATE_HANDSHAKE_IN_PROGRESS,
    QUIC_STATE_ESTABLISHED,
    QUIC_STATE_DRAINING,
    QUIC_STATE_CLOSED_FINAL
} quic_connection_state_t;

typedef struct {
    quic_connection_state_t state;
    uint64_t next_packet_number;
    uint64_t largest_acked;
    uint64_t peer_packet_number_space;
    uint8_t connection_id[QUIC_MAX_CONNECTION_ID];
    uint8_t connection_id_len;
    bool handshake_complete;
} quic_connection_t;
```

该状态机的关键迁移：
- `CLOSED` → `INITIAL_SENT`：客户端发送 `Initial`
- `INITIAL_SENT` → `HANDSHAKE_IN_PROGRESS`：收到服务器 `Handshake` 或 `Retry`
- `HANDSHAKE_IN_PROGRESS` → `ESTABLISHED`：完成 TLS 握手并收到 `Finished`
- `ESTABLISHED` → `DRAINING`：发送或接收 `CONNECTION_CLOSE`
- `DRAINING` → `CLOSED_FINAL`：等待结束期限后释放连接

#### 8.14 协议交互补充细节
- `CRYPTO` 帧在 Initial、Handshake 和 0-RTT 阶段分别使用不同加密密钥保护。
- `Version Negotiation` 包仅在客户端与服务器版本不兼容时出现。
- 服务器可以在 1-RTT 阶段通过 `NEW_CONNECTION_ID` 改变连接 ID，以支持负载均衡和迁移。
- `PATH_CHALLENGE` / `PATH_RESPONSE` 用于确认地址可达性，并在迁移前验证新路径。

### 9. 变长整数编码（Variable Length Integer）
- QUIC 使用可变长度整数来减少头部大小。长度为 1、2、4、8 字节。
- 编码规则：最高两位表示长度。
  - `00`：1 字节，总长度 1
  - `01`：2 字节，总长度 2
  - `10`：4 字节，总长度 4
  - `11`：8 字节，总长度 8
- 例如：
  - `0x25` 表示 `37`，1 字节编码。
  - `0x7b34` 表示 `0x3b34`，2 字节编码。
- 典型用途：Stream ID、Offset、Length、Packet Number Length、ACK ranges、Transport Parameters 等。
- 优点：小值占用更少字节，节省包头开销；大值仍可编码。

### 9. Packet Number 截断与解码规则
- QUIC 报文只发送截断的 Packet Number（1-4 字节），以减少包头大小。
- 接收端需要恢复完整 Packet Number：
  1. 读取当前包中的截断 `packet_number`。
  2. 计算一个 `expected` 值，通常是上一个已接收最大包号 + 1。
  3. 根据截断长度构造一个候选范围，使得恢复后的包号最接近 `expected`。
- 公式示例：
  - `window = 1 << (8 * pn_length)`
  - `candidate = (expected & ~(window - 1)) | truncated_pn`
  - 如果 `candidate + window/2 <= expected`，则 `candidate += window`
  - 如果 `candidate > expected + window/2`，则 `candidate -= window`
- 目的：在不发送完整 64-bit 包号的前提下，保持包号单调递增，满足丢失检测。
- 结果：同一连接内，可正确还原真实包号，避免重放攻击和包号回绕问题。

### 10. 其他重要编码规则
- 长头部和短头部包中的 `Packet Number Length` 同样以 2 位字段表示，可取 1~4 字节。
- `Length` 字段在长头部包中使用可变整数编码，表示当前包中 `Packet Number` + `Payload` 的总长度。
- 许多帧类型都使用可变整数字段，比如 `STREAM` 帧中的 `Stream ID`、`Offset`、`Length`；`ACK` 帧中的 `Largest Acknowledged`、`ACK Delay`、`ACK Range Count` 等。

#### 11.1 Initial 包示例
```c
static const uint8_t initial_payload[] = {
    /* 这里假设已经加密的 CRYPTO+STREAM 负载 */
};
```
